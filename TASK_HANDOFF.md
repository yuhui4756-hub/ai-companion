# 任务交接规则

所有专业线程完成任务后，使用下面格式写交接，方便总控决定下一步交给谁。

详细流程见 `docs/COLLABORATION_FLOW.md`。

## 关键机制

- 背景专业线程不会自动把最终结果推送到总控线程。
- 总控派发任务后，必须记录目标线程 ID，并在预计完成后主动使用线程读取工具回收结果。
- 专业线程的最终回复必须包含“交接记录”；总控回收后负责把重要结论写入 `docs/HANDOFF_LOG.md`。
- 专业线程的最终回复必须包含“给下一个线程的任务包”；如果暂时没有下一线程，写明“建议交回总控”。
- 如果目标线程明确且不涉及用户决策/高风险操作，专业线程应把任务包发送给目标线程，并在交接记录中写明“已发送给：线程名/线程ID”。
- 如果无法发送、目标不明确或涉及风险，专业线程必须交回总控，由总控决定下一步。
- 专业线程发送任务包后必须停止当前任务，不等待、不轮询、不追踪接收线程。
- 总控不持续盯线程过程，只在需要阶段结果、返工结论或最终提交时回收。
- 如果专业线程完成但没有按格式交接，总控要主动补齐交接摘要，并把该问题记录到阶段日志。
- 总控不得在专业线程仍在执行同一任务时并行实现；只有线程失败、明确阻塞，或总控先终止任务后，才可接手。
- 总控不得替代测试验收线程做中间验收；开发产物必须先交给测试验收线程。
- 总控的最终确认只包括：交接是否完整、构建是否通过、仓库是否干净、是否存在真实密钥、是否可以提交推送。

## 失败退回规则

| 测试发现的问题 | 退回线程 |
| --- | --- |
| 页面报错、功能不可用、构建失败、数据保存失败 | 开发实现 |
| 伴侣语气冷、油腻、越界、情绪价值不足 | AI人设设计 |
| 功能范围不清、用户流程不合理、优先级冲突 | 产品经理 |
| 模块边界混乱、接口设计不稳、扩展路径错误 | 技术架构 |
| 交接信息缺失、文档不一致、线程状态混乱 | 记录交接 |
| 需要用户登录、填 Key、确认发布或付费 | 总控 |

## 标准交接格式

```md
## 交接记录

- 交接来源：
- 建议交接给：
- 当前阶段：
- 完成内容：
- 修改文件：
- 验证方式：
- 遗留问题：
- 需要用户参与：
- 下一步建议：
```

## 下一个线程任务包格式

```md
## 给下一个线程的任务包

- 目标线程：
- 任务类型：
- 背景：
- 必读文件：
- 输入材料：
- 具体任务：
- 不要做：
- 验收标准：
- 完成后交接给：
```

## 交接判断

| 当前问题 | 交给谁 |
| --- | --- |
| 不知道该做什么功能 | 产品经理 |
| 聊天冷冰冰、不像伴侣 | AI人设设计 |
| 不知道系统怎么拆 | 技术架构 |
| 需要写代码或修 bug | 开发实现 |
| 不确定是否达标 | 测试验收 |
| 文档散乱、上下文丢失 | 记录交接 |
| 需要用户登录、授权、确认风险 | 总控 |

## 交接记录要求

- 交接必须说明当前任务是否完成。
- 交接必须列出修改文件、验证方式和遗留问题。
- 交接必须给出明确下一步对象，不能只写“继续推进”。
- 交接必须说明任务包是否已经发送给下一个线程；如果没有发送，要说明原因。
- 交接发送后，本线程不要等待对方完成；等待总控或用户的新任务即可。
- 如果需要返工，必须写明退回线程、问题类型、复现方式和期望结果。
- 如果需要用户参与，必须说明用户只需要做什么，不要把专业判断丢给用户。

## 总控对用户提问规则

总控只在以下情况向用户提问：

- 需要登录、验证码、授权或付费确认。
- 需要用户选择明显影响方向的产品定位。
- 涉及删除数据、上传隐私、公开发布、付费接口调用等高风险操作。
- 本地环境或第三方平台需要用户亲自操作。

其他专业问题默认由对应线程自行查证、判断并推进。
