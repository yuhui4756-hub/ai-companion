# MVP 产品规格

## 第一版目标

做出一个可运行的中文网页 Demo，让用户可以：

- 选择或轻量自定义一个 AI 伴侣。
- 填写自己的 API Key。
- 和伴侣持续聊天。
- 让伴侣提供情绪价值和基础问题解答。
- 查看、编辑、删除长期记忆。

## 用户流程

1. 用户打开网页。
2. 用户选择伴侣类型和默认聊天风格。
3. 用户填写 API 服务商、模型、API Key 和接口地址。
4. 用户进入聊天界面。
5. 系统根据伴侣人设、用户画像、长期记忆组织提示词。
6. 用户聊天过程中，系统自动识别可沉淀的长期记忆。
7. 用户可进入记忆管理页查看、编辑、删除记忆。

## 伴侣类型

- 温柔朋友：稳定倾听、鼓励、日常陪伴。
- 轻恋爱陪伴：更亲密，但避免过度成人化。
- 理性支持：帮用户拆问题、做计划、给建议。
- 治愈陪伴：语气柔和，适合压力、低落、焦虑时使用。
- 角色扮演：允许动漫感或设定感，但仍遵守安全边界。

## MVP 功能优先级

### P0 必须有

- 单页网页聊天界面。
- 伴侣类型选择。
- 用户自填 API Key。
- OpenAI 兼容接口调用。
- 本地长期记忆存储。
- 记忆查看、编辑、删除。
- 基础错误提示，如 API Key 错误、网络失败、模型返回失败。

### P1 重要但可后置

- 多供应商预设，如 OpenAI、DeepSeek、通义千问、Kimi、智谱等。
- 伴侣头像和名称自定义。
- 用户首次问卷。
- 聊天记录搜索。
- 导入/导出配置。

### P2 后续增强

- QQ/微信入口适配。
- 多设备同步。
- 云端账号系统。
- 伴侣市场或角色模板分享。
- 语音输入输出。

## 非目标

- 第一版不承诺真实接入 QQ/微信。
- 第一版不做医疗、心理治疗或法律等高风险专业服务。
- 第一版不保存或代管用户商业 API Key 到远程服务器。

## 工程化升级阶段规格

这一阶段服务于“把现有项目升级成更完整的 AI 应用开发简历项目”，不替换已经完成的恋爱陪伴主体验。

### P0 必须完成

- 本地后端代理：前端通过本机接口请求模型，由代理统一处理 OpenAI 兼容接口、错误码和跨域问题。
- 数据访问收敛：伴侣、消息、长期记忆、风格摘要不再散落在 UI 中直接读写存储，至少有清晰的 repository/service 边界。
- SQLite 方案落地或明确替代方案：桌面版核心数据具备比 `localStorage` 更可靠的本地持久化路径。
- 旧数据迁移：已有 `localStorage` 数据可以导出或迁移，不能让用户升级后误以为数据丢失。
- 最小知识库/RAG：用户可以导入文本资料，系统能本地保存、检索，并在聊天时只注入相关片段。
- 隐私说明更新：清楚说明 API Key、聊天记录、长期记忆和导入资料在哪里保存、什么时候会发给模型服务商。

### P1 重要但可后置

- 模型请求流式输出。
- 多供应商预设进一步拆分和健康检查。
- 知识库来源管理、删除、重新索引。
- 简单检索命中解释，帮助用户知道回复参考了哪些导入资料。
- 桌面版迁移向导和备份恢复入口。

### P2 后续增强

- embedding 向量检索。
- 多设备加密同步。
- 插件化模型供应商适配。
- QQ/微信等外部入口的合规接入。

### 本阶段验收口径

- 面试表达上能讲清楚：前端、后端代理、本地数据库、记忆、知识库检索和模型请求之间的边界。
- 普通用户使用上不能明显变复杂；恋爱伴侣聊天仍是第一屏主体验。
- 所有隐私敏感数据默认本地保存，任何外发都必须是用户主动配置的模型请求所需内容。
